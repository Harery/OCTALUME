## 1. Design Principle: Progressive Disclosure

The skill is NOT a monolithic file. It uses a three-tier loading system:

```
Tier 1 — Metadata (always in context, ~100 tokens)
  name + description in YAML frontmatter
  Used for: trigger detection only

Tier 2 — Parent orchestrator (loads on trigger, ~290 lines)
  SKILL.md — routing table, pyramid rules, constraints summary,
  enhancements index, gotchas, approval gate protocol

Tier 3 — Active stage sub-skill (loads on demand, 31–79 lines each)
  stages/s01-intake.md through stages/s12-maintenance.md
  Only ONE loads at a time — the currently active stage
  Reference files load only when their enhancement executes
```

**Why this matters:** Loading all 17 files simultaneously would flood the
context window with ~1,300 lines, causing attention dilution and instruction
drift by Stage 6. Progressive disclosure keeps the active footprint under
300 lines at any moment.

## 3. Pyramid Questioning System

Questions narrow as the pipeline descends. The layer determines question type,
not just question count.

```
S1–2   MACRO    Big picture · vision · users · problem · market
S3–4   MESO     Business model · revenue · competition · priorities
S5–6   MICRO    Features · flows · constraints · stack decisions
S7–8   NANO     Technical specifics · architecture · security · logic
S9     ATOMIC   Diagrams derived from all prior stages (no new questions)
S10–12 BUILD    Generate · monitor · maintain (no new questions at build layer)
```

Hard rules:
- 1–3 questions max per stage boundary. Never more.
- Assume inferable gaps. Flag them. Never ask about them.
- If unknown → search the internet. Never guess.

## 4. Five Operating Rails

These govern every recommendation, every tool choice, every architectural
decision across all 12 stages. Violation = Silent Firewall failure.

| Rail | Rule |
|---|---|
| AI-first | 90–95% autonomous. Operator translates intent only. No dev team. |
| Ubuntu LTS | Only server OS. Verify Ubuntu compatibility before recommending anything. |
| Zero-cost | Free/OSS until operator pre-approves paid. Always. |
| Live search | Search before any tool, version, pricing, or SEO answer. |
| SEO day one | First-class requirement. Stages 1 through 12. Public pages always. |

## 6. Context Compaction Checkpoint

Location: after Stage 5 approval, before Stage 6 loads.
Purpose: prevent context overflow on long pipeline runs.

Protocol:
1. Generate Session Handoff Summary (E5 format)
2. Recommend `/compact` to operator
3. Re-inject: project name, product type, key decisions, approved stages
4. Load Stage 6 sub-skill in clean context

Without this: by Stage 7–8, the context carries 5 stages of outputs plus
conversation history. Automatic compaction loses fine-grained Stage 1–3
decisions that Stage 8 logic definitions need to trace back to.

## 7. Silent Firewall — 34 Checks

Run mentally before every delivery. Fix failures silently. Customer never sees this.

**Questionnaire (5):** Step 0 opening sent · dump received before questions · 9 dynamic questions from actual gaps · each question has 4 choices + recommended + custom · all 9 answered before S1 output

**Quality (6):** readable output · deferred items explained · no jargon ·
assumptions flagged · partner voice · pyramid layer correct

**Integrity (6):** no hallucinated versions · search-verified tools ·
decisions trace to BRD · SpecKit standard · logic traces S1–3 · Mermaid renderable

**Constraints (4):** Ubuntu LTS compatible · no unauthorised paid tools ·
SEO on public pages · AI operable confirmed

**File outputs (6):** all 9 docs present before S10 · CLAUDE.md ≤100 lines ·
MEMORY.md ≤200 lines · constitution.md principles-only · HANDOFF.md current ·
cost-ledger.md written at S6

**Token (3):** single-stage sub-skill loaded · compaction after S5 · CLAUDE.md uses @ imports

**Enhancements (4):** E2 classifier applied · E1 CLAUDE.md generated ·
E6 cost ledger current · E5 handoff written

## 8. Approval Gate Protocol

After every stage output, present the gate:
> "That's Stage [N] — [Name]. Does this match your vision?"

- Yes → load next stage sub-skill, proceed
- Revise → adjust output and re-present before proceeding
- Skip → warn of downstream gaps, proceed only with explicit operator approval

## 10. Gotchas (Known Failure Modes)

Built from predictive analysis. Expand from real project runs.

1. AI tends to over-scope MVP — always push back to single core loop
2. Stage 6 drifts toward paid tools without explicit cost tier enforcement
3. CLAUDE.md grows past 100 lines by Stage 10 — enforce curation at Stage 12
4. Stage 4 competitor search returns aggregator sites — verify original sources
5. Stage 8 logic blocks multiply fast on complex projects — scope to MVP only
6. E5 session handoff is skipped under time pressure — it must not be optional

Add new gotchas here after every real project run.

## 11. What This Skill Is NOT

- Not an agent (yet). It is a passive instruction set. The operator is the
  orchestration layer. See the skill-vs-agent analysis for upgrade path.
- Not complete without real runs. The Gotchas section will grow. Eval test
  cases will be added. The skill improves from field data, not design sessions.
- Not immutable. Every real failure adds a gotcha. Every real project adds a
  test case. Update the skill after every 3–5 real runs.

## 12. Upgrade Path — Skill to Agent

The skill is already architecturally agent-ready (Option 3):

```
Today (skill):
  Operator pastes Brain Dump → Claude runs pipeline → operator approves gates

Future (agent using this skill):
  Operator pastes Brain Dump → agent loads s01-intake.md → runs stage →
  writes output to file → surfaces gate to operator → loads s02 on approval →
  runs through all 12 stages autonomously → final handoff to Claude Code CLI
```

Do NOT convert the skill into the agent (Option 2). Keep them separate.
The skill is the brain. The agent is the hands. Build the agent after the
first 1–2 real project runs prove the pipeline end-to-end.
